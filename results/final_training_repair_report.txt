MultiModal TaskGate — Final Training Repair Report
===================================================
Generated: 2026-03-15

1. BUGS DETECTED & FIXED
-------------------------

A) NaN Loss (escalating through epochs)
   Root Cause: Label mapping gaps — compound labels like "neutral,positive",
   "very_positive", "not harmful" were unmapped, fell through to float()
   conversion which failed, producing -1. But the deeper NaN came from
   entropy_regularization() in losses.py: gate values in fp16 could exceed
   [0,1] range, causing log(negative) → NaN.
   Fix:
     • mm_dataset.py: Complete rewrite with exhaustive label maps per task,
       compound-label splitting (take first token before comma), and strict
       per-task validation. Unknown labels → -1 (masked in loss).
     • losses.py: entropy_regularization() now casts gate values to float32
       and clamps to [1e-6, 1-1e-6] before log(). total_loss initialized as
       a proper tensor on the correct device.

B) NaN Gradient Norms
   Root Cause: clip_grad_norm_ was called on ALL model.parameters()
   including frozen params with None gradients → returned nan.
   Also, fp16 overflow in backward is normal AMP behavior — GradScaler
   should skip those steps, not crash.
   Fix:
     • clip_grad_norm_ now called only on trainable params.
     • GradScaler overflow detection: compare scale before/after step.
       If scale decreased → overflow event (step skipped), counted but
       not treated as error.

C) Learning Rate Hardcoded at 1e-4
   Root Cause: All stages used lr=1e-4.
   Fix: Stage-specific LRs: stage1=2e-5, stage2=1e-5, stage3=5e-6.

D) Windows Multiprocessing Crash
   Root Cause: num_workers=2 triggers multiprocessing.spawn on Windows,
   which re-imports the entire module in child processes.
   Fix: Set num_workers=0 (inline loading, no subprocess spawning).

E) Low GPU Memory Reporting (~0.6 GB)
   Root Cause: torch.cuda.memory_allocated() only counts explicit tensor
   allocations, not cached/reserved memory.
   Fix: Changed to torch.cuda.max_memory_reserved() which reports true
   VRAM footprint (expected 3-5 GB).
